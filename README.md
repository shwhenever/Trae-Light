# TRAE 红绿灯 — 工作状态监视器
最近看到了Vibe Coding，给Trae写了一个，
一开始想支持TraeIDE和TraeWork的，TraeIDE支持好像有Bug，TraeWork没问题。

始终置顶在屏幕最前方的小窗，用红绿灯实时反映 **TRAE Work / TRAE IDE** 的工作状态。做别的工作时也能一眼看到 TRAE 当前是在空闲，还是在干活。

**零依赖**：只使用系统自带的 Python 3.12、tkinter 和 ctypes，不安装任何第三方包，双击即用。进程检测与 CPU 采样全部走 Windows 原生 API，即使没有 psutil 也能正常工作。
实在没有，可以使用打包好的。

## 状态含义

三盏灯垂直排列在右侧灯箱里，只有当前状态对应的灯亮起，其余两盏呈暗色填实。灯箱和展开面板均为直角矩形。

| 灯 | 含义 | 判定依据 |
|----|------|----------|
| 红  | TRAE 未运行 | 未检测到 TRAE 进程 |
| 绿| 运行中，空闲 | 检测到进程，且最近无真实工具调用 |
| 黄| 工作中 | 检测到进程，且最近有真实工具调用或生成活动 |


## 使用方式

默认出现在屏幕右上角，透明背景，只占一小条竖向区域。把鼠标移上去会向屏幕内侧展开详情面板，移开后自动收起。

- **悬停**：展开详情，显示产品名、状态原因、进程数与主进程、内存、系统 CPU、最近活动时间、会话与任务 ID、最近事件列表。
- **拖动**：按住左键拖动可移动到任意位置，松开后位置写入 `pos.json`，下次启动恢复。
- **单击**：把 TRAE 窗口提到最前。
- **右键**：弹出菜单，包含“将 TRAE 置前”“重置到屏幕右上角”“关于”“退出”。
- **单实例**：再次双击启动器不会开第二个窗口，已运行的实例会继续工作。

## 判定原理
感兴趣的可以看一下。

判定基于一个更精确的信号：**距上一条“真实工作行”的秒数**。真实工作行特指日志里带有实际执行工具或生成动作标记的行，包括 `invoke_via_toolhost`（调用工具）、`Tool call completed`（工具调用完成）、`ExecuteCommand request`（执行命令请求）、`call_server_generate_plan`（调用服务端生成）、`emit_intermediate`（生成中间输出）、`generate_plan_item`（生成计划项）。这些标记只在 agent 真正处理请求时出现，轮询心跳不会产生。

主信号加上一个副信号：最近 `RATE_WINDOW` 秒内的日志行数。当真实工作行距今不超过 `WORK_ACTIVE_THRESH` 秒，或者最近窗口内行数超过 `ENTER_RATE`，就判为工作中（黄）。进入工作状态后，需要持续 `WORK_GRACE` 秒没有上述活动才回到空闲（绿），这层滞回用来覆盖工具调用之间的短暂间隙，避免灯光频繁抖动。

TRAE 的 AI agent 把执行过程写到：

```
\<TRAE 数据目录>\logs\<时间戳>\Modular\ai-agent_*_stdout.log
```

工具会自动发现最新的会话日志目录并跟踪它，无需手动配置路径。

## 系统占用率

详情面板里的“系统 CPU”最初显示为 0，原因是 psutil 的非阻塞 `cpu_percent` 首次调用没有基准可比较，按其约定返回 0.0。工具改用 Windows 原生 `GetSystemTimes` API 自己实现了跨调用保持基准的采样器，用 idle、kernel、user 三个时间值的差计算占用率。首次采样建立基准返回 0，之后每次都返回真实值。

## 文件说明

- `trae_monitor.py` — 状态监测逻辑，包含进程检测、日志分析、CPU 采样、状态判定。可单独运行 `python trae_monitor.py` 做自检，会连续采样三次并打印状态、速率、原因、最近事件。
- `trae_traffic_light.py` — 红绿灯界面，基于 tkinter Canvas 绘制，负责置顶透明窗口、悬停展开、呼吸动画、拖动、右键菜单。
- `Start.bat` — 启动器，用系统 `pythonw.exe` 无控制台窗口启动。
- `pos.json` — 记忆窗口位置，首次运行后生成。

## 运行环境

需要 Python 3.12 及自带的 tkinter，默认路径 `C:\Program Files\Python312\pythonw.exe`。若安装在其他位置，可修改 `启动红绿灯.bat` 里的路径，或直接在脚本目录下运行：

```
pythonw.exe trae_traffic_light.py
```

## 调参

所有可调参数集中在 `trae_monitor.py` 顶部。

- `WORK_ACTIVE_THRESH`：真实工作行距今多少秒内判为工作中，默认 6 秒。
- `RATE_WINDOW` / `ENTER_RATE`：最近窗口时长与行数阈值，默认 3 秒内满 22 行也判为工作，用于快速触发。
- `WORK_GRACE`：进入工作后，非活跃持续多久才回空闲，默认 3 秒。
- `TRAE_PROCESS_PATTERNS`：识别 TRAE 进程的进程名子串，默认 `trae solo` 和 `trae`。
- `POLL_MS`（在 `trae_traffic_light.py`）：状态刷新间隔，默认 1500 毫秒。

如果发现该变绿时还亮黄，说明 TRAE 后台仍在跑真实工具调用（这是 TRAE 自身的行为，不是判定 bug），可以把 `WORK_ACTIVE_THRESH` 调小试试。如果反过来，该亮黄时却变绿，说明工具调用间隔超过了阈值，可以适当调大。

## 常见问题

**为什么我没在思考，灯还是黄的？**

TRAE 的 ai-agent 是一个共享进程，红绿灯监视的是整个进程的活动，而不是单独某一路对话。即使你这边没有在交互，只要后台有真实工具在跑（比如同步会话状态、处理之前的尾迹、其他面板触发的任务），灯就会显示工作状态。这是符合预期的——它反映的是 TRAE 当前是否在干活，而不是你这一路对话是否在思考。
